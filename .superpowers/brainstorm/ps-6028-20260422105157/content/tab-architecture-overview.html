FishMark 标签页设计 · 架构总览

已选方向：C. 主进程拥有全局工作区图。这里先确认职责边界和主数据流，不展开菜单和关闭弹窗细节。

核心分层
main: WorkspaceGraph 持有全局窗口、标签页、标签顺序、活动标签、标签归属窗口、拖拽迁移状态。

这里是“FishMark 当前打开了哪些文档、它们在哪个窗口里”的唯一事实源。
main: TabSession Registry 每个标签页一条 session：路径、标题、当前草稿内容、上次已保存内容、dirty、保存状态、外部文件变更状态。

关闭标签、拖到别的窗口、切换活动标签，都不丢会话。
renderer: Window Shell + Editor Host 只负责把当前窗口的 tab strip、active tab、编辑器视图和提示 UI 渲染出来。

CodeMirror 仍在 renderer，本地处理输入和 IME；通过受限 IPC 把变更提交给 main session。
主数据流
1. 任何“打开 / 新建 / 关闭 / 拖出成新窗口”命令先到 main main 决定是复用当前窗口、往某个窗口插入标签、还是新建窗口承接标签。
2. main 更新 WorkspaceGraph 与 TabSession 然后把“某窗口当前有哪些标签、活动标签是谁、每个标签的轻量状态”同步给对应 renderer。
3. renderer 只挂载当前活动标签的编辑器 切换标签时，renderer 请求 main 给出该 tab session 的最新草稿内容和元数据，再重载编辑器。
4. 编辑器内容变化回写到 main renderer 产生内容变化、dirty 状态、保存请求后，通过受限 IPC 更新对应 tab session；main 再广播 tab strip 所需的标题/dirty/save 状态。
这条路的取舍
好处 标签拖出成新窗口、后续会话恢复、最近焦点窗口、跨窗口移动标签都会有统一模型，不需要再从单窗口逻辑里硬拆。
代价 这轮必须正式引入 workspace / tab IPC 协议，`currentDocument` 心智模型会升级成 `window -> tabs -> active tab session`。
我建议保留的边界 不把原始文件系统能力下放给 renderer；不把 CodeMirror 挪到 main；不在本轮加入启动恢复。
