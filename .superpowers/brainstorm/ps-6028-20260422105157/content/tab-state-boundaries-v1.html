FishMark 标签页设计：状态边界

第二段设计：明确 main 和 renderer 分别持有什么状态，以及标签切换 / 拖到新窗口时怎样保持未保存内容、光标和编辑器体验。

main 持有的状态
WorkspaceGraph
windowId -> tabIds -> activeTabId
还包括标签顺序、拖拽中的源窗口 / 目标窗口、最近焦点窗口。
TabSession
path、name、draftContent、lastSavedContent、dirty、saveState、externalFileState。
未保存内容在这里是稳定真值。
TabViewSnapshot
轻量视图快照：当前 selection、滚动锚点、阅读/编辑模式。
用于切回标签或拖到新窗口后尽量恢复现场。
Watcher / Close Flow
外部文件监听、保存/放弃/取消关闭的决策流程都由 main 编排，避免窗口关闭和标签关闭走两套逻辑。
renderer 持有的状态
CodeMirror 实例
真实输入、撤销重做、IME 组合输入、DOM focus 都仍在 renderer。
我们不把编辑器搬到 main。
窗口级 UI 状态
设置页开关、拖拽 hover 样式、局部 toast、当前 DOM 布局细节。
这些不需要成为全局工作区真值。
组合输入中的临时态
composition 中间态不上传到 main 做复杂协调。
renderer 只在稳定变更点把最新文本快照推送给 main。
标签栏视图
renderer 只消费 main 发来的窗口快照来渲染 tab strip，不自己推导全局工作区结构。
同步策略
1. 不做 patch 协议，先做版本化文本快照同步
renderer 在文档变化后把完整文本和版本号推给 main。这样先保住实现清晰度，不引入操作流重放。
2. 发送节奏是“微批量”，不是每个按键都立即 round-trip
renderer 本地即时更新编辑器；main 接收经过短节流的最新草稿快照。这样拖标签、关标签、保存时都能拿到近实时真值。
3. 标签切换前先 flush 当前标签快照
切换标签、拖出到新窗口、关闭窗口前，renderer 先把当前标签的最新文本和 view snapshot 同步到 main，再执行结构变更。
