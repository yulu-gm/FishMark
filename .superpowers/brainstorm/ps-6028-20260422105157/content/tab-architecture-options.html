A. Renderer 直接扩成多标签页

改动看起来最少，但会把现在围绕 currentDocument 的逻辑全部塞进更大的 renderer 状态树里。

优点：起步快
风险：App.tsx 会继续膨胀

B. 窗口工作区模型

把“窗口”和“标签页”拆成两层状态，保留现有文件 IO 边界；新窗口只处理窗口壳，文档会话仍主要在 renderer 内。

推荐：边界最清晰
适合先做完整多标签 + 拖出新窗口
Main Session Manager 主进程掌控所有窗口与标签
Main 维护全局 workspace graph 窗口、tab、最近焦点、拖拽迁移全在主进程
Renderer 更像 tab 的视图层和编辑宿主
IPC 需要大量增量协议和同步事件

C. Main 拥有全局工作区图

最适合未来会话恢复、多窗口编排和复杂恢复逻辑，但这轮会引入明显更大的 IPC 和主进程状态复杂度。

优点：长期扩展性最好
代价：这轮实现面过大

我的当前建议

优先走 B。它能把现有单文档模型自然演进成“窗口工作区 + 多个文档会话”，同时不把文件系统、保存、外部变更监听重新上提到主进程。这样最容易保住现有架构边界，也足够支撑你要的排序和拖出成新窗口。
