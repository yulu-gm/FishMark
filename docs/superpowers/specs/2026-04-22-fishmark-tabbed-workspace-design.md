# FishMark 标签页工作区设计

## Goal

为 FishMark 引入默认单窗口多标签页的工作区模型，使应用可以像 VS Code 一样在同一窗口内打开多个 Markdown 文档；只有用户明确通过 `File > New Window` 时才打开新窗口。

本轮设计同时覆盖：

- 当前窗口内打开、切换、关闭多个 Markdown 标签页
- `File > New` 在当前窗口创建新的未保存标签页
- `File > Open...`、外部打开文件、拖入文件时默认进入当前窗口标签页
- 标签页拖拽排序
- 标签页拖出为新窗口，以及跨窗口移动标签
- 未保存标签页在关闭标签或关闭窗口时的逐个确认流程

本轮不包含会话恢复：应用重启后不恢复上次的标签页或多窗口布局。

## Scope

### In scope

- 主进程中的全局工作区状态模型
- 标签页级文档会话模型
- `main / preload / renderer` 之间新增的工作区 IPC 协议
- 当前窗口标签栏 UI 与活动标签切换逻辑
- 拖拽排序与拖出成新窗口的结构语义
- 现有打开 / 保存 / autosave / 外部文件变更链路向 `tabId` 维度迁移
- 受影响的设计、验收、测试与进度文档更新

### Out of scope

- 应用重启后的工作区恢复
- 多标签页历史导航、分屏编辑、固定标签页、预览标签页
- 最近文件列表的产品入口扩展
- 远程协作或多进程共享同一文档会话
- 将 CodeMirror 或编辑输入逻辑迁移到主进程

## Product decisions

### 1. 默认工作方式

- 空白欢迎页窗口打开第一个文档时，原地进入该文档
- 当前窗口已经有文档标签时，再次 `Open...`、外部打开文件、拖入文件，默认在当前窗口新建标签页并切换过去
- 只有用户明确触发 `File > New Window` 时才创建新窗口

### 2. New 与 New Window 的职责分离

- `File > New`：在当前窗口创建新的未保存标签页 `Untitled.md`
- `File > New Window`：创建新的空窗口，不自动创建文档标签

### 3. 关闭未保存标签页的语义

- 关闭单个标签页时，只处理该标签页自己的未保存状态
- 关闭整个窗口时，按窗口中的标签页逐个处理未保存项
- 不使用“汇总全部未保存标签页的一次性总弹窗”

### 4. 本轮能力边界

本轮一步到位支持：

- 标签页切换
- 标签页关闭
- 标签页拖拽排序
- 标签页拖出为新窗口
- 跨窗口移动标签

但仍保持单窗口只挂载一个活动编辑器实例，不同时保活多个 CodeMirror 视图。

## Architecture

### 1. 总体方向：main 拥有全局工作区图

本轮采用长期可扩展的工作区架构：由 `main` 持有全局工作区真值，而不是让每个 renderer 只维护本窗口的局部状态。

理由：

- 标签拖出成新窗口和跨窗口移动标签，本质上是窗口边界内外的结构迁移，不适合由 renderer 自治再让主进程补洞
- 后续如果增加会话恢复、最近焦点窗口、窗口布局恢复，主进程全局真值可以直接承接
- 可以避免把当前单文档 renderer 状态不断膨胀成一个更大的 UI 状态树

### 2. 核心模型

#### WorkspaceGraph

`WorkspaceGraph` 由 `main` 持有，描述整个应用当前的工作区结构：

- 哪些窗口存在
- 每个窗口下有哪些标签页
- 标签页顺序
- 每个窗口的活动标签页
- 拖拽中的标签页来源、目标和插入位置
- 最近焦点窗口

`WorkspaceGraph` 是“窗口与标签结构”的唯一事实源。

#### TabSession

每个标签页对应一个 `TabSession`，也由 `main` 持有。它是“文档会话状态”的唯一事实源，至少包含：

- `tabId`
- `path`
- `name`
- `draftContent`
- `lastSavedContent`
- `isDirty`
- `saveState`
- `externalFileState`
- `encoding`

其中 `draftContent` 必须保存在 `main`，而不是只留在 renderer。这样在以下场景下可以稳定转移未保存内容：

- 拖出为新窗口
- 跨窗口移动标签
- 关闭当前窗口前逐个处理未保存标签页
- 后续潜在的会话恢复

#### TabViewSnapshot

为了在标签切换或拖入新窗口后尽量恢复现场，`main` 还保存轻量视图快照：

- 当前 selection
- 滚动锚点
- 阅读 / 编辑模式

它不是 Markdown 真值，也不是编辑器内部真值，只是标签页切换体验的恢复辅助数据。

### 3. renderer 的职责

renderer 不再持有整个文档会话真值，而是做三件事：

- 渲染当前窗口的标签栏与窗口壳层
- 挂载当前活动标签的 CodeMirror 编辑器
- 将本地编辑行为通过受限 IPC 同步回 `main`

CodeMirror 继续留在 renderer，原因是：

- IME 组合输入、DOM focus、选区、浏览器输入事件仍然最适合在 renderer 层处理
- 主进程不应拥有富编辑器 DOM 和输入细节
- 继续保持 `main / preload / renderer` 的安全边界

## State boundaries

### main 持有

- `WorkspaceGraph`
- `TabSession Registry`
- `TabViewSnapshot`
- 关闭流程编排状态
- 文件监听与外部变更状态

### renderer 持有

- 当前活动标签的 CodeMirror 实例
- 组合输入过程中的临时状态
- 窗口局部 UI 状态：设置页开关、hover、toast、拖拽视觉反馈等
- 来自主进程窗口快照的只读投影视图

### 保持的边界

- 不把文件系统访问权限暴露给 renderer
- 不把 CodeMirror 挪到 `main`
- 不让 renderer 推导全局工作区结构

## Sync strategy

### 1. 版本化文本快照，而不是 patch 协议

本轮同步策略优先清晰性与可靠性：renderer 向 `main` 提交版本化文本快照，而不是操作级 patch 或事务重放。

优点：

- 更容易与现有单文档模型对接
- 更容易保证拖拽迁移与关闭确认时拿到稳定文本真值
- 不会在第一版标签页架构里额外引入复杂操作日志系统

### 2. 微批量同步

renderer 本地编辑器继续即时更新，不等待主进程回写。同步到 `main` 的草稿文本采用短节流的微批量策略：

- 保持主线程输入体验稳定
- 让 `main` 足够快地拿到近实时草稿真值
- 减少每个按键都 round-trip 的噪音

### 3. 结构变更前强制 flush

在以下结构变更发生前，renderer 必须先把当前活动标签的最新草稿与视图快照 flush 到 `main`：

- 切换标签
- 关闭标签
- 拖拽排序完成
- 拖出为新窗口
- 关闭窗口

这样可以确保结构迁移和关闭判断基于同一份最新会话数据。

## Command and IPC design

### 1. 所有结构命令先到 main

以下命令统一先由 renderer 发到 `main`，由 `main` 决定工作区结构变化：

- 新建标签
- 打开文件
- 打开文件路径
- 新建窗口
- 激活标签
- 关闭标签
- 重排标签
- 拖出标签为新窗口
- 跨窗口接收标签
- 请求关闭窗口

### 2. 打开路径的统一规则

所有文件打开入口收敛到同一条主进程决策链：

- `File > Open...`
- 启动参数带文件路径
- 系统级“用 FishMark 打开”
- 拖入 Markdown 文件

决策规则：

- 如果目标窗口当前没有文档标签：原地承接第一个文档
- 如果目标窗口已有文档标签：在当前窗口新增标签并激活
- 如果用户明确触发 `New Window`：新建空窗口

### 3. 关闭行为

关闭标签：

- `main` 读取该 `tabId` 的 `TabSession`
- 若未保存，走单标签确认流程
- 若保存 / 放弃成功，再从 `WorkspaceGraph` 删除该标签

关闭窗口：

- `main` 枚举该窗口的标签顺序
- 对 dirty 标签逐个走相同的单标签确认流程
- 全部处理完成后才真正关闭窗口

### 4. 保存链路按 tabId 定位

现有这些行为从“当前唯一文档”改为“指定 `tabId` 的文档会话”：

- `saveMarkdownFile`
- `saveMarkdownFileAs`
- autosave
- 外部文件变更检测
- 关闭前保存判断

## UI and editor lifecycle

### 1. 单窗口只保活一个活动编辑器

每个窗口同一时刻只挂载当前活动标签的一个 CodeMirror 实例。非活动标签不保活隐藏编辑器，只保留：

- `TabSession`
- `TabViewSnapshot`

原因：

- 降低 IME、focus、shortcut、autosave 并发风险
- 避免多个编辑器实例在同窗口内同时竞争状态
- 让第一版多标签页的复杂度集中在工作区结构，而不是编辑器池管理

### 2. 切换标签的恢复语义

切换到某个标签时：

- renderer 请求该 `tabId` 的最新 `TabSession`
- 用 `draftContent` 重载活动编辑器
- 尽量恢复 `TabViewSnapshot` 中的 selection / scroll / shell mode

这保证标签页在切换后尽量回到离开前的工作现场，但不承诺像后台保活编辑器那样保留完整浏览器内部状态。

### 3. 标签栏只渲染轻量状态

标签栏由窗口快照驱动，显示的信息只包含轻量可视状态：

- 标签标题
- dirty 标记
- 保存中状态
- 当前活动标签

标签栏不直接读取本地编辑器内部状态，不在 renderer 内生成全局工作区推论。

## Testing

### 必须覆盖的测试层

- `src/main/`
  - 工作区结构变更
  - 新建窗口 / 新建标签
  - 标签拖出成新窗口
  - 关闭窗口时逐标签未保存处理
- `src/preload/`
  - 新增工作区 IPC 契约
- `src/renderer/`
  - 标签栏交互
  - 活动标签切换
  - flush 后切换 / 关闭 / 拖拽
  - 单窗口只挂载一个活动编辑器的约束

### 回归重点

- autosave 不因标签切换丢失内容
- 外部文件变更提示按正确 `tabId` 命中
- 新建未保存标签与已有保存文档混合时，关闭流程仍稳定
- 拖出为新窗口后，未保存草稿内容与 dirty 状态不丢
- IME 基线不因切换标签或重载活动编辑器被破坏

## Risks

- 主进程状态面显著扩大，若 IPC 设计不清晰，容易引入多处镜像状态
- 当前大量逻辑围绕 `currentDocument` 编写，迁移到 `tabId` 维度会带来较大重构面
- 拖出为新窗口涉及窗口生命周期与编辑器卸载重挂，容易影响 dirty/flush 时机
- 若文本同步节流设计不当，可能导致关闭窗口或拖拽时丢失最后一次输入
- 单窗口只挂载一个编辑器实例虽然更稳，但会让标签切换时的恢复逻辑更关键

## Acceptance

- 当前窗口可以同时打开多个 Markdown 标签页
- `File > New` 在当前窗口创建新的未保存标签页
- `File > Open...`、拖入文件、外部打开文件在已有文档时默认新增标签页
- `File > New Window` 才会显式创建新窗口
- 标签页支持排序与拖出为新窗口
- 关闭单个标签页时只处理该标签页的未保存状态
- 关闭窗口时逐个处理该窗口中的未保存标签页
- 不破坏现有保存、autosave、外部文件变更、IME 与 Markdown round-trip 基线
