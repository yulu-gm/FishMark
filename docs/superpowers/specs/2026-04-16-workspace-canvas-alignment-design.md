# Workspace Canvas Alignment Design

日期：2026-04-16

## 背景

当前 editor shell 已经具备 `rail + workspace + settings drawer` 的基本骨架，但在默认浅色主题下，`workspace` 内仍混用了两套对齐逻辑：

- 顶部信息带按工作区左边界展开
- 空态卡片和文档编辑区按自己的居中宽度单独排布

结果是品牌信息、右上提示和中间主卡片没有共享一条稳定的视觉轴线，界面看起来像几块彼此独立的 demo 区块，而不是一个成熟的桌面写作工具。

同时，当前状态条仍附着在文档流底部。文档变长后，用户需要滚到最底才能看到保存状态和字数，这不符合写作工具对“持续可见状态反馈”的预期。

## 目标

本轮设计解决两个直接问题：

1. 统一 `workspace` 的版心逻辑，采用“顶部信息带 + 独立居中画布”的布局模型。
2. 将状态条从文档内容流中抽离，改为应用底部固定状态条。

附带目标是顺手把默认浅色主题收敛到更明确的气质：安静、干净、写作工具、苹果系质感。

## 非目标

- 不新增任何导航、最近文件或大纲能力
- 不重写 editor state、autosave、theme runtime 或 preferences 模型
- 不引入新的主题 manifest 或设计 token 结构
- 不把整个 header 并入主画布，避免丢掉应用级信息带

## 备选方案

### 方案 A：整个 workspace 统一成一条内容列

顶部 header、空态卡片、文档头、编辑器和状态条全部共享一条主列。

优点：
- 对齐规则最简单
- 实现后最整齐

缺点：
- 顶部信息带会失去“应用级”而更像“页面正文”
- 右侧提示文案的呼吸感会明显下降

### 方案 B：顶部信息带 + 独立居中画布

保留横向顶部信息带；下方空态卡片、文档头和编辑器共用同一条居中画布列。

优点：
- 既保留桌面应用头部气质，也保证主画布稳定
- 打开文档前后结构连续，视觉切换自然
- 最符合当前 Yulora 的写作工具定位

缺点：
- 需要更明确地区分“应用级宽度”和“画布级宽度”

### 方案 C：维持现有 header，只修空态卡片偏移和状态条位置

优点：
- 改动最小

缺点：
- 顶部与主画布之间的轴线关系仍然模糊
- 只是修补，不是真正建立稳定基线

## 选型

采用方案 B。

原因是它在“桌面应用壳层感”和“单栏写作画布稳定性”之间取得了最好的平衡，也与用户确认的方向一致。

## 布局设计

### 1. 总体结构

`app-layout` 继续保留两列：

- 左侧 `rail`
- 右侧 `workspace`

`workspace` 内改成三层：

1. `workspace-header`
2. `workspace-canvas`
3. `app-status-bar`

其中：

- `workspace-header` 是应用级信息带
- `workspace-canvas` 是文档级画布容器
- `app-status-bar` 是固定底栏，不再属于文档流

### 2. 顶部信息带

顶部仍保留品牌信息和提示文案的横向布局，但不再和下方卡片抢同一套“正文版心”规则。

设计要求：

- 宽度与 `workspace` 外边距一致
- 左右两端保持克制的桌面应用留白
- 品牌块左对齐，提示文案右对齐
- 纵向高度压缩到辅助信息级别，不再像主内容标题区

### 3. 独立居中画布

空态、文档态统一进入 `workspace-canvas`，共用一条稳定的居中列。

这条列负责：

- 空态欢迎卡片
- 文档头
- 编辑器外框

规则：

- 画布列始终居中
- 画布列有明确 `max-width`
- 空态卡片可以比编辑器外框更“完整”，但它的左边界必须和文档头、编辑器外框共线
- 文档打开后不再出现 `document-bar`、`document-canvas`、`document-editor` 各自定义宽度导致的偏移

### 4. 固定底部状态条

状态条提升为应用级底栏，固定在 `workspace` 下方。

内容维持三段：

- 左侧：保存状态
- 中部：字数
- 右侧：平台信息

要求：

- 始终可见
- 不跟随文档滚动
- 与画布区之间用轻边界和浅背景分层
- 编辑区底部必须预留足够空间，避免最后一行内容被遮挡

## 默认浅色主题方向

本轮不重做主题系统，但会细化 `default-light` 的视觉 token 和表层样式，让它更贴近以下方向：

- 页面背景：浅灰白，不做重色块分割
- 设置抽屉：半透明白玻璃，边界清晰，模糊克制
- 主卡片与编辑器：纯白底，轻阴影，圆角稳定
- 文字：深蓝灰而不是纯黑
- 强调色：少量冷色，用于 focus、活动状态和轻提示

要避免的视觉噪音：

- 过重的渐变
- 过蓝或过紫的科技感背景
- 抽屉和卡片使用两套完全不同的材质语言
- rail、workspace、card 之间边界过多导致碎片化

## 交互与滚动

### 设置抽屉

设置抽屉保留当前左侧滑入模型，但视觉上更偏“玻璃层覆盖在应用之上”，而不是另一页设置。

要求：

- 打开时不卸载编辑器
- 点击遮罩、点击关闭按钮、按 `Esc` 都可关闭
- 关闭后恢复原有焦点语义

### 文档滚动

文档滚动区域仍由编辑器自身承担，外层只负责容器和底部空间。

要求：

- 固定状态条不能截断编辑器最后几行内容
- 空态不应出现额外滚动条
- 文档态在较长内容下仍保持 header、画布和底栏职责清晰

## 测试与验收

自动验证至少覆盖：

- `workspace` 新结构的关键 DOM 标识
- 设置抽屉仍能打开/关闭
- 状态条存在且不再挂在文档流内部
- autosave、theme runtime、preferences 不回归

人工验收重点：

1. 空态下，顶部信息带与居中卡片关系自然，不再“歪”
2. 打开文档后，文档头、编辑器外框与空态卡片共用同一条画布轴线
3. 长文档滚动时，底部状态条始终可见
4. 状态条不会遮挡文档末尾内容
5. 默认浅色主题观感符合“安静、干净、苹果系写作工具”

## 风险

- 结构层与主题层边界如果重新混淆，后续主题切换还会再次破坏布局
- 固定底栏如果没有预留底部内边距，会直接影响可编辑区域
- 如果仍保留多套 `max-width` 规则，视觉偏移只会换一种形式继续存在

## 实施提示

实现时优先做这三件事：

1. 收敛 `workspace`、`document-bar`、`document-canvas`、`document-editor` 的宽度规则
2. 先把状态条从文档流中抽离，再补滚动安全区
3. 最后再细调默认浅色主题 token 与抽屉材质
