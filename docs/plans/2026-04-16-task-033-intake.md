# TASK-033 Intake

Task: `TASK-033`
Goal: Add the first stable fenced code block rendering path so non-active code fences read like rendered code, while active code fences remain raw Markdown and preserve info strings, indentation, newlines, and round-trip safety.

In scope:
- extend `packages/markdown-engine` block-map types with top-level fenced code block support
- capture each fenced code block's info string and exact source range from `micromark`
- keep fenced code blocks inside the existing `src/renderer/code-editor.ts` decoration pipeline
- style non-active fenced code blocks with monospace typography, preserved whitespace, horizontal overflow handling, and hidden fence markers
- keep active fenced code blocks in raw Markdown source form with no alternate data model
- preserve existing editor semantics for Tab, Shift+Tab, Enter, IME composition, undo/redo, and autosave by not adding new input handlers
- add regression coverage for parser output and CodeMirror inactive/active rendering transitions

Out of scope:
- syntax highlighting
- inline code rendering
- language-aware editing behaviors
- nested block decomposition inside code fences
- renderer-side React widgets or a second document representation
- changes to `main`, `preload`, save bridges, or autosave policy

Landing area:
- `packages/markdown-engine/src/block-map.ts`

Acceptance:
- fenced code blocks appear in `parseBlockMap()` output in source order with exact offsets, line ranges, and preserved info string
- moving the cursor outside a code fence shows a non-active rendered state with hidden fences, monospace styling, and preserved line breaks/indentation
- moving the cursor back into the code fence restores full Markdown source immediately, including opening/closing fences and info string
- saving after editing a fenced code block keeps the original Markdown structure round-trippable
- existing Enter behavior, IME composition guard, and block rendering for heading/paragraph/list/blockquote remain intact

Verification:
- `npm run test -- packages/markdown-engine/src/parse-block-map.test.ts`

- `npm run typecheck`

- `npm run build`

Risks:
- code fence decorations span multiple lines, so using the wrong decoration type can break whitespace preservation or cursor mapping
- parser offsets must stay aligned with CodeMirror-normalized document text, especially when documents originated with CRLF line endings
- any custom key handling inside code fences could regress Tab/Enter or IME stability, so this slice should stay decoration-only
