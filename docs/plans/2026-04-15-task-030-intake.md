# TASK-030 Intake

> 设计文档以中文 spec `docs/plans/2026-04-15-task-030-cli-driven-design.zh-CN.md` 为准。

Task: TASK-030
Goal: 重新交付基于 CLI 的真实编辑器测试执行能力，让 CLI 负责启动 Yulora editor 测试窗口并执行所有编辑器内操作，同时让 workbench 同步显示当前步骤与每一步结果。
In scope:
- 明确首版 synthetic gradient 方案的回退边界，并以此作为本轮重做的起点
- 让 `packages/test-harness` 的 CLI 成为真实 editor 场景的唯一执行入口
- 让 CLI 启动或复用真实的 Yulora editor 测试窗口，并执行所有编辑器内操作
- 在 `packages/test-harness` 中补齐真实步骤执行器，使步骤不再停留在 headless placeholder
- 让 workbench 从 CLI 运行会话订阅或消费步骤事件，而不是本地直接跑 runner
- 让 workbench 基于 CLI 的真实执行结果同步显示当前步骤、步骤状态、失败原因与终态
- 为新的 editor-driven 执行路径补齐最小自动化测试覆盖
Out of scope:
- 截图、baseline、diff 图与相关版本管理
- 一次性扩展大批新场景
- 脱离现有 Electron / React / test-harness 架构做宽泛重构
Landing area:
- `src/main/`
- `src/preload/`
- `src/renderer/`
- `packages/test-harness/src/cli/`
- `packages/test-harness/src/handlers/`
- `packages/test-harness/src/scenarios/`
- `packages/test-harness/src/runner.ts`
- 相关 Vitest / workbench 测试
- `MVP_BACKLOG.md`
- `docs/progress.md`
- `reports/task-summaries/TASK-030.md`
Acceptance:
- `TASK-030` 不再依赖 synthetic gradient、截图或伪 visual 结果作为默认验收路径
- agent 可通过统一 CLI 入口启动真实 editor 场景
- CLI 会打开或复用真实的 Yulora editor 测试窗口，并执行所有编辑器内操作
- scenario step 能驱动真实 editor 行为，并把每一步结果反馈给 CLI / runner
- workbench 能展示来自 CLI 运行会话的当前步骤、已完成步骤、失败原因和终态，且不回退既有 debug 能力
Verification:
- `npm run test -- packages/test-harness/src/cli/run.test.ts`
- `npm run test -- src/main/runtime-windows.test.ts`
- `npm run test -- src/renderer/test-workbench.test.tsx`
- `npm run test -- packages/test-harness/src/runner.test.ts`
- `npm run lint`
- `npm run typecheck`
- `npm run build`
Risks:
- 如果 CLI 与 editor window 之间没有受限而稳定的命令桥，真实步骤执行会停留在口号层
- 如果步骤执行结果不能稳定映射回 runner 事件，debug 面板会失去可信度
- 需要避免破坏 `TASK-025` 的窗口隔离边界、`TASK-028` 的 debug 面板与 `TASK-029` 的 CLI 契约
- 需要避免向 renderer 暴露不受限制的 Node API
