# TASK-008 Intake

Task: `TASK-008`
Goal: Implement the first real Markdown-engine capability by parsing Markdown with `micromark` and producing a stable top-level block map that later active-block and block-rendering tasks can consume.

In scope:
- define the minimal block-map types under `packages/markdown-engine/`
- add a `micromark`-backed parse entrypoint
- emit `heading`, `paragraph`, `list`, and `blockquote` blocks in source order
- preserve stable source offsets and line ranges for each block
- add unit tests for common inputs and boundary cases
- extend repo typecheck/test coverage so `packages/markdown-engine` is part of the gates

Out of scope:
- renderer integration
- active-block selection
- block rendering
- nested list item/task list decomposition
- inline token mapping
- code fence, image, table, and link block support

Landing area:
- `packages/markdown-engine/src/block-map.ts`
- `packages/markdown-engine/src/parse-block-map.ts`
- `packages/markdown-engine/src/index.ts`
- `packages/markdown-engine/src/parse-block-map.test.ts`
- `package.json`
- `tsconfig.vitest.json`
- `tsconfig.renderer.json`
- `vite.config.ts` only if test discovery needs adjustment

Acceptance:
- `parseBlockMap()` returns top-level blocks for `heading`, `paragraph`, `list`, and `blockquote`
- block order matches source order
- heading depth and list ordered/unordered metadata are exposed
- block offsets can recover exact original Markdown slices
- unit tests cover representative success and edge cases

Verification:
- `npm run test -- packages/markdown-engine/src/parse-block-map.test.ts`
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`

Risks:
- touches Markdown round-trip foundations, so offsets and exact source slicing must stay correct
- parser/test config may drift if `packages/markdown-engine` is still outside current repo gates
- downstream tasks depend on this shape, so type overdesign would create avoidable coupling
